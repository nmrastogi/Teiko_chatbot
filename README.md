## Quick start (GitHub Codespaces / local)

```bash
# 1. Install dependencies
make setup

# 2. Set your Anthropic API key
cp .env.example .env
# Edit .env and add your key: ANTHROPIC_API_KEY=sk-ant-...

# 3. Run the full data pipeline (creates teiko.db + outputs/)
make pipeline

# 4. Launch the chatbot dashboard
make dashboard
```

The app opens at `http://localhost:8501`.

## Database schema

### Design

Three normalized tables:

```sql
subjects(subject_id PK, project, condition, age, sex, treatment, response)
samples(sample_id PK, subject_id FK, sample_type, time_from_treatment_start)
cell_counts(id PK AUTOINCREMENT, sample_id FK, b_cell, cd8_t_cell, cd4_t_cell, nk_cell, monocyte)
```

### Rationale

**Why normalize?**  
The source CSV repeats subject-level metadata (project, condition, age, sex, treatment, response) on every sample row. With 3 samples per subject this is manageable, but at scale — hundreds of projects, thousands of subjects — denormalization causes data anomalies: updating a subject's response field requires touching every row for that subject. Three tables prevent that.

**subjects** stores one row per patient. Any update to subject metadata (response adjudication, age correction) touches one row.

**samples** links each biological sample to its subject and records when it was collected. New sample types (e.g., tumor biopsies, bone marrow) are added as rows, not columns — no schema migration needed.

**cell_counts** holds the measurements. Keeping counts separate from sample metadata means new assay panels (more cell types) can be accommodated by adding columns here without touching the other tables. Alternatively, a fully EAV `(sample_id, population, count)` model could support unlimited populations at the cost of wider pivot queries.

**Indexes** on `samples.subject_id` and `cell_counts.sample_id` make joins fast even at thousands of samples.

**Scalability path:**  
At hundreds of projects / thousands of samples, add:
- A `projects` table (project metadata, PI, institution) → foreign key in subjects.
- Partitioning `cell_counts` by project for analytical query performance.
- A `populations` lookup table if the panel grows — allows adding populations without schema changes.
- A `timepoints` table if the visit schedule becomes complex.

## Code structure

| File | Role |
|------|------|
| `load_data.py` | Standalone script. Drops and recreates all three tables, then bulk-loads `cell-count.csv` via pandas `to_sql`. Idempotent — safe to re-run. |
| `analysis.py` | Standalone script. Connects to `teiko.db`, runs all queries, produces four output artifacts in `outputs/`. Separated from the chatbot so the pipeline can be run headlessly (`make pipeline`) without a browser. |
| `chatbot.py` | Streamlit app. On startup it checks for `teiko.db` and `outputs/` and bootstraps them if missing, so the app is self-contained for deployment. Sends pre-computed results in the Claude system prompt as context; gives Claude a `query_database` tool for ad-hoc SQL. |

**Design principle:** The pipeline (`load_data.py` → `analysis.py`) is pure data — no UI dependency. The chatbot layer is pure presentation — it reads artifacts but doesn't recompute them. This makes both independently testable and keeps Streamlit's caching from interfering with long-running DB operations.

## Deploying to Streamlit Community Cloud

1. Push this repository to GitHub.
2. Go to [share.streamlit.io](https://share.streamlit.io) → **New app**.
3. Select your repo, branch `main`, main file `chatbot.py`.
4. Under **Advanced settings → Secrets**, add:
   ```toml
   ANTHROPIC_API_KEY = "sk-ant-..."
   ```
5. Click **Deploy**. On first load the app auto-runs the pipeline.
6. Paste the public URL into the dashboard link at the top of this README.
